Vamos detalhar o Nível 2: Testes de Cenário (Avaliação Offline). Este é o passo mais importante para garantir que a sua nova arquitetura de busca seja objetivamente melhor e mais "humana".
O objetivo é criar um "gabarito" para o seu sistema de RAG. Assim, você pode medir o desempenho de qualquer alteração (como ajustar o threshold, mudar o modelo de embedding, ou refinar a estratégia de chunking) de forma consistente e automática.
Passo a Passo para Criar e Usar o Conjunto de Avaliação
Passo 1: Selecionar Documentos de Referência
Escolha um ou dois documentos representativos da sua base de conhecimento. O "Relatório Gerencial - Manatí" é um ponto de partida perfeito. É crucial que você conheça bem o conteúdo desses documentos para criar o gabarito.
Passo 2: Criar a Tabela de Avaliação (o "Golden Set")
Esta tabela é o coração do processo. Vamos usar uma planilha ou um arquivo CSV para organizá-la. Ela terá as seguintes colunas:
question_id: Um identificador único para cada pergunta (ex: MANA_001).
question_text: O texto da pergunta, simulando a linguagem natural do usuário.
question_type: Uma categoria para garantir que estamos testando diversos cenários.
expected_chunk_ids: A lista de IDs dos chunks do ChromaDB que, juntos, contêm a resposta ideal para a pergunta. Este é o "gabarito".
ideal_answer_summary: Um resumo em uma frase da resposta que um humano daria, para referência.

Observações importantes sobre a tabela:
Mapeamento é Crucial: A parte mais trabalhosa é preencher a coluna expected_chunk_ids. Você precisará inspecionar seu ChromaDB para encontrar os IDs corretos para cada pergunta.
Diversidade de Perguntas: Note como as perguntas variam de forma e intenção. Isso é fundamental para testar a robustez do sistema. A MANA_006 ("e sobre os cotistas?") é um exemplo avançado que testaria a memória da conversa, mas por enquanto podemos focá-la em testar se uma busca genérica por "cotistas" funciona.
Passo 3: Escrever o Script de Avaliação
Agora, você precisa de um script em Python que automatize a avaliação.
Python
import chromadb # ou o seu cliente de DB
# Suponha que você tenha sua lógica de busca encapsulada em uma função
from your_rag_system import enhanced_search 

# 1. Carregue sua tabela de avaliação (ex: de um arquivo CSV)
evaluation_set = [
    {"question_id": "MANA_001", "question_text": "fale sobre o mana11", "expected_chunk_ids": ['chunk_1', 'chunk_2', 'chunk_4']},
    {"question_id": "MANA_002", "question_text": "qual foi o dividendo do mana11 em 2025?", "expected_chunk_ids": ['chunk_15']},
    # ... resto das perguntas
]

results = []
# 2. Itere sobre cada pergunta no conjunto de avaliação
for item in evaluation_set:
    question = item["question_text"]
    expected_ids = set(item["expected_chunk_ids"])
    
    # 3. Execute sua função de busca para obter os chunks recuperados
    # Suponha que a função retorne uma lista de documentos (com IDs)
    retrieved_docs = enhanced_search(query=question, top_k=5) 
    retrieved_ids = set([doc.id for doc in retrieved_docs])
    
    # 4. Calcule as métricas
    is_hit = len(expected_ids.intersection(retrieved_ids)) > 0
    
    # Precision@K: Dos recuperados, quantos são relevantes?
    precision_at_k = len(expected_ids.intersection(retrieved_ids)) / len(retrieved_ids) if retrieved_ids else 0
    
    # Recall@K: Dos relevantes, quantos foram recuperados?
    recall_at_k = len(expected_ids.intersection(retrieved_ids)) / len(expected_ids) if expected_ids else 0
    
    results.append({
        "question_id": item["question_id"],
        "is_hit": is_hit,
        "precision": precision_at_k,
        "recall": recall_at_k
    })

# 5. Agregue e imprima os resultados
total_hits = sum([r["is_hit"] for r in results])
hit_rate = total_hits / len(results)
avg_precision = sum([r["precision"] for r in results]) / len(results)
avg_recall = sum([r["recall"] for r in results]) / len(results)

print(f"Resultados da Avaliação (com top_k=5):")
print(f"Hit Rate: {hit_rate:.2%}")
print(f"Average Precision: {avg_precision:.2%}")
print(f"Average Recall: {avg_recall:.2%}")

# Você também pode imprimir os resultados por pergunta para depurar falhas

Como Usar Isso para Melhorar a Busca
Agora você tem um sistema de medição. O processo de melhoria se torna um ciclo científico:
Medir (Baseline): Execute o script com a sua arquitetura de busca atual. Anote os resultados. Este é o seu ponto de partida.
Modificar: Implemente a mudança proposta pelo agente (a busca híbrida com filtro de metadados).
Medir Novamente: Execute o mesmo script de avaliação na nova arquitetura.
Comparar: Compare os novos resultados (Hit Rate, Precision, Recall) com a sua baseline. Os números melhoraram? A Precision aumentou para perguntas com ticker? O Recall melhorou para perguntas abertas?
Ao analisar as falhas (is_hit = False), você pode ajustar sua estratégia. Talvez precise recuperar mais chunks (top_k=7 em vez de 5), ou talvez o chunking de uma seção específica precise ser refeito.
Este método transforma a melhoria da "qualidade da busca" de um conceito abstrato em um problema de engenharia mensurável e iterativo.
