## Resumo do problema a resolver

PDFs de apresentação misturam texto, gráficos e tabelas. A extração atual transforma tudo em texto corrido; tabelas perdem estrutura; imagens/legendas ficam desconectadas. Resultado: chunks sem contexto e respostas erráticas.

---

## 1) Pipeline recomendado (alto nível — implementável no app)

1. Upload no app → arquivo salvo no storage (fonte de verdade).
2. Pré-processamento de página (converter cada página em imagem de alta qualidade).
3. Extração paralela multimodal:

   * Texto estruturado via OCR/layout-aware (ou extração nativa de PDF quando possível).
   * Extração de tabelas para formato tabular (CSV/JSON), preservando cabeçalhos e coordenadas.
   * Extração de imagens/gráficos (salvar imagem + bounding box).
   * Extração de legendas e texto próximo ao gráfico (associar por proximidade espacial).
4. Normalização e conversão: transformar toda saída em unidades semânticas (document → seções → blocos textuais → tabelas estruturadas → imagens com legenda).
5. Chunking sensível à estrutura (ver regras abaixo): nunca quebrar tabela no meio; tratar tabela inteira como chunk + um resumo textual.
6. Enriquecimento de metadados (tipo, tickers, página, posição, responsável, vigência, confiança).
7. Vetorização:

   * Texto: embeddings textuais.
   * Tabelas: representar em duas formas (a) embedding do resumo textual da tabela + (b) embedding tabular via serialização controlada.
   * Imagens/gráficos: embedding visual (CLIP-like) e optionally OCR text embedding da legenda.
   * Combinar (concat/ensemble) vetores multimodais quando pertinente.
8. Indexação no Vector DB (ChromaDB ou equivalente) com metadados indexáveis.
9. QA humano (interface para revisar/extrair/editar tabelas e legendas) antes de marcar “ativo para retrieval”.
10. Retrieval híbrido (vetor + filtros metadata) → re-rank → síntese LLM.

(Arquitetura: upload UI → queue workers para passo 2–7 → indexação → Stevan usa retrieval e LLM para responder.)

Referência sobre chunking e impacto: Chroma research sobre estratégias de chunking. ([research.trychroma.com][1])

## 2) Regras de extração e chunking (para garantir “busca perfeita”)

### Extração por página

* Para cada página gere: texto bruto, layout (posições), lista de tabelas com coordenadas, lista de imagens com coordenadas, bounding boxes de títulos/subtítulos.

### Tabelas

* Detectar tabelas com bibliotecas especializadas (Camelot/Tabula) ou serviços (Document AI / Textract) para obter CSV/JSON.
* **Nunca** transformar tabela em texto corrido no index final. Sempre armazenar:

  * Tabela original (CSV/JSON).
  * Resumo textual curto (3–6 frases) que explica linhas/colunas mais relevantes e as tendências.
  * One-line header string (para matching).
* Criar um chunk exclusivo para a tabela (com metadados: page, bbox, tipo_chunk=table, tickers_relacionados).
* Se uma tabela for muito grande, fragmentá-la por lógica de sub-tabelas (blocos de linhas coerentes), preservando cabeçalhos em cada fragmento.

### Gráficos / imagens

* Extrair imagem em alta resolução + OCR da região próxima (legenda).
* Gerar embedding visual e também embedding do texto extraído das legendas.
* Criar chunk do tipo image_chart com campos: resumo_textual_do_grafico (uma frase com insights), eixoX/eixoY quando detectável, período, fonte, confiança.

### Texto corrido / slides

* Agrupar por seções visuais (título + bloco associado). Evitar cortes no meio de ponto explicativo.
* Tamanhos: para texto normal, 400–800 tokens; para tabelas usar a tabela inteira como um chunk independentemente do token count (ou fragmentar logicamente).

### Metadados obrigatórios por chunk

* documento_id, página, tipo_chunk (text|table|image_chart), tickers_relacionados, categoria, status, data_validade, fonte, confiança_extract.

Essas regras reduzem “chunks quebrados” e preservam semântica de tabelas/figuras.

## 3) Ferramentas / modelos recomendados por etapa (pragmático)

### Extração OCR / Layout

* Nível produção (se orçamento/privacidade permitirem): Google Document AI ou AWS Textract para extração layout-aware e tabelas robustas. Esses serviços extraem tabelas e formas com boa qualidade e oferecem best practices. ([Google Cloud][2])
* Alternativa open source: combinar pdfplumber / PyMuPDF (fitz) para texto + Tesseract OCR em imagens de página para casos onde o PDF é “imagem”. pdfplumber é bom para extração textual; Camelot / Tabula para tabelas quando elas são bem formatadas. Camelot costuma funcionar bem em tabelas com linhas; Tabula em casos de stream. ([camelot-py.readthedocs.io][3])

### Extração de tabelas

* Primeiro tentou Camelot/Tabula/pdfplumber (local) para muitos uploads automáticos.
* Para casos falhos, usar Document AI / Textract para recuperação mais confiável; idealmente rodar validação paralela.

### Modelos layout-aware / multimodal

* Donut / LayoutLM family para interpretar o layout e extrair entidades e relações diretamente da página-imagem (útil para slides complexos com imagens e texto combinados). Donut pode ser usado para parsing sem OCR tradicional em alguns cenários. ([GitHub][4])

### Embeddings e multimodal

* Texto: embeddings OpenAI (ou outro provedor) para texto.
* Imagem: embeddings CLIP ou modelo similar (para gráficos/imagens). Combine image embeddings + text embeddings (por exemplo, concatenação ou re-ranker ensemble).
* Use estratégia multimodal RAG para recuperar tanto chunks textuais quanto imagens e permitir re-ranking multimodal. Guias práticos recentes mostram Bom ganho no uso de embeddings visuais + textuais. ([Medium][5])

### Vector DB

* ChromaDB funciona bem; mantenha ability to store metadata and multimodal embeddings. Garanta versioning e tombstones para documentos expirados. (Você já usa Chroma; mantenha e ajuste esquema de metadados.) ([Emergent Mind][6])

## 4) Estratégia de vetorização multimodal e combinação

* Para cada chunk crie:

  * embedding_text (se chunk tiver texto)
  * embedding_image (se tiver imagem/plot)
* Modo de armazenamento:

  * salvar ambos embeddings e metadados
  * opcionalmente salvar um embedding combinado para buscas rápidas
* Retrieval:

  * passo 1: filtro metadata (status=ativo, tickers match se query tem ticker, etc.)
  * passo 2: busca vetorial híbrida (texto + imagem) — executar duas buscas e unir resultados por score normalizado
  * passo 3: re-ranking por heurística (prefer tabelas quando query for numérica; prefer gráficos quando query envolver tendências visuais)
* Na síntese LLM: enviar primeiro as tabelas (CSV/summary) e legendas relevantes, depois blocos textuais; indicar claramente ao LLM o tipo de cada bloco (TABLE / IMAGE / TEXT) e qual metadado chave (page, source, confidence).

## 5) UX / IHM e QA humana (essencial)

Para garantir ingestão perfeita no app (Replit frontend + backend):

1. Upload guiado: obrigue preenchimento de metadados (tipo, tickers, estratégia, vigência).
2. Preview automático pós-extração: mostrar ao usuário — a) texto extraído, b) tabelas detectadas com botão “Editar CSV”, c) imagens detectadas com legenda extraída — permitir correção inline.
3. Painel de aprovação: só após aprovação manual o documento entra como “ativo” para vetorização.
4. Playground de teste: responsável escreve uma pergunta e vê os chunks que Stevan recuperaria (com scores) e a resposta simulada. Dá feedback imediato.
5. Histórico e versioning: controlar versões e permitir rollback; marcar obsoletos.
6. Alerts automáticos: conteúdos próximos de expirar e conteúdos que sofreram muitas transferências humanas (indicador de pobre qualidade).

## 6) Regras práticas para LLM prompt (uso do conteúdo tabular)

Quando enviar tabelas para LLM:

* Nunca injetar tabela cru longuíssimo; preferir:

  * header + top-N linhas relevantes + resumo textual e conclusões
  * ou CSV anexo curto + “Resuma em 3 pontos as conclusões quantitativas”
* Se pergunta é numérica (“qual foi variação do yield?”), priorize chunks do tipo table com número, inclua contexto de data.
* Sempre inclua metadado: fonte, página, confidence.

## 7) Boas práticas operacionais (versão curta)

* Vetorização só para conteúdo aprovado.
* Versionar embeddings: quando documento muda, criar nova versão e tombstone da antiga no vector DB.
* Monitorar recuperação: logar quais chunks foram usados e se geraram transferência humana; usar métricas para re-treinar regras de chunking/summary.
* Tratar tabelas como first-class citizens: sempre armazenar CSV+summary+embedding.

Referência sobre best practices para table extraction e queries: AWS Textract best practices e Document AI guidelines. ([AWS Documentation][7])

## 8) Ferramentas/methods recomendados (prático)

* PDF → imagens: PyMuPDF / pdf2image
* Texto extraction: pdfplumber / PyMuPDF + Tesseract OCR (para imagens)
* Table extraction: Camelot / Tabula local + fallback para Document AI / Textract for hard cases. ([camelot-py.readthedocs.io][3])
* Layout/semantic parsing: Donut / LayoutLM for complex slide parsing. ([GitHub][4])
* Image embeddings: CLIP variants
* Text embeddings: OpenAI (ou provedor on-prem if required)
* Vector DB: ChromaDB (com schema para multimodal embeddings) ([Emergent Mind][6])
